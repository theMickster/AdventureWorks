# Epic 1.3: HR CRUD UI

**Initiative**: [Initiative 1 - Sales + HR Management Web Application](../initiatives/01-sales-hr-web-application.md)
**Timeline**: Weeks 5-6
**Estimated Story Points**: 100-120
**Status**: Not Started

---

## Epic Overview

**As an HR manager, I want to manage employees and departments through a web interface, so that I can create, view, update, and list these entities efficiently with proper validation and error handling.**

---

## Epic Success Criteria (Gherkin Format)

**Given** the HR CRUD UI is complete
**When** I use the application
**Then** I should be able to:
- ✅ View a paginated list of employees
- ✅ Create a new employee with form validation
- ✅ Update an existing employee
- ✅ View employee details
- ✅ View a list of departments
- ✅ Create a new department
- ✅ Update an existing department
- ✅ View department details
- ✅ See loading states and error messages

---

## Features

### Feature 1.3.1: Employee List View
**As a user, I want to see a paginated list of all employees.**

**Acceptance Criteria**: Display table with Name, Job Title, Department, Hire Date, pagination support.
**API Endpoint**: `GET /api/hr/employees?pageNumber=1&pageSize=10`
**Estimated Story Points**: 8

---

### Feature 1.3.2: Employee Create Form
**As a user, I want to create a new employee with validation.**

**Acceptance Criteria**: Form fields for First Name, Last Name, Job Title, Department (dropdown), Birth Date, Hire Date, Gender, Marital Status.
**API Endpoint**: `POST /api/hr/employees`
**Estimated Story Points**: 13

---

### Feature 1.3.3: Employee Update Form
**As a user, I want to update an existing employee.**

**Acceptance Criteria**: Pre-filled form with current data, validation, successful update.
**API Endpoint**: `PUT /api/hr/employees/{id}`
**Estimated Story Points**: 13

---

### Feature 1.3.4: Employee Detail View
**As a user, I want to view detailed information about an employee.**

**Acceptance Criteria**: Display all employee information, department details, employment dates.
**API Endpoint**: `GET /api/hr/employees/{id}`
**Estimated Story Points**: 8

---

### Feature 1.3.5: Department List View
**As a user, I want to see a list of all departments.**

**Acceptance Criteria**: Display table with Department Name, Group Name, buttons for create/edit.
**API Endpoint**: `GET /api/hr/departments`
**Estimated Story Points**: 8

---

### Feature 1.3.6: Department Create Form
**As a user, I want to create a new department.**

**Acceptance Criteria**: Form fields for Department Name, Group Name, validation.
**API Endpoint**: `POST /api/hr/departments`
**Estimated Story Points**: 13

---

### Feature 1.3.7: Department Update Form
**As a user, I want to update an existing department.**

**Acceptance Criteria**: Pre-filled form, validation, successful update.
**API Endpoint**: `PUT /api/hr/departments/{id}`
**Estimated Story Points**: 13

---

### Feature 1.3.8: Department Detail View
**As a user, I want to view detailed information about a department.**

**Acceptance Criteria**: Display department info, list of employees in department.
**API Endpoint**: `GET /api/hr/departments/{id}`, `GET /api/hr/employees?departmentId={id}`
**Estimated Story Points**: 8

---

### Feature 1.3.9: TypeScript Models for HR
**As a developer, I want TypeScript interfaces for Employee and Department.**

**Acceptance Criteria**: Interfaces match backend DTOs exactly.
**Key Files**: `libs/hr/data-access/src/lib/models/employee.model.ts`, `libs/hr/data-access/src/lib/models/department.model.ts`
**Estimated Story Points**: 5

---

### Feature 1.3.10: Form Validation for HR
**As a developer, I want frontend validation matching backend rules.**

**Acceptance Criteria**: Required fields, date validations, string length limits match backend.
**Backend Reference**: `Application/HumanResources/Employees/Commands/CreateEmployee/CreateEmployeeCommandValidator.cs`
**Estimated Story Points**: 8
